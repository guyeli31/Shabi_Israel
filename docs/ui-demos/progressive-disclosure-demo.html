  • One long scrolling form — admin can't see the player picker without scrolling past identity fields.

  • Review tab aggregates all validation; Create button stays disabled until green.

Issue #2 — Sticky first col

  • League table on mobile has 9 columns + a 360px viewport. Any horizontal scroll right makes the Player column slide off-screen.
  • Once player names are gone, the numeric grid is meaningless — you can't tell whose 5.04 Luck you're looking at.
  • The codebase already has stickyCols support in the MF format — it's just not wired for production league pages.

The fix

  • Pin the Player column with position: sticky; inset-inline-start: 0. Background must be opaque so cells below don't bleed through.
  • Use a JS scroll listener (debounced) on the wrap to add .is-scrolled-x when scrollLeft > 0 — that triggers the shadow.
  • v2 mapping: the existing stickyShadow.js module lives in src/tables/MFTable/. Auto-enable it for all D/E presets on viewports < 768px.
  • For RTL pages this works automatically because inset-inline-start is logical-property, not left.

Issue #6 — Tile-row overflow on narrow viewports

  • Dashboard hero renders 3-4 tiles in a single flex row. On iPhone SE (320px), the third tile's content clips horizontally.
  • Player names (which sit in the "Leading" tile) are unpredictable in width — "Y" is short, "Danny_kondrea" is long. Fixed widths can't accommodate.
  • This is an anti-pattern: the user sees half a name and assumes the data is corrupted.

The fix

  • Replace flex-row with CSS Grid auto-fit: grid-template-columns: repeat(auto-fit, minmax(140px, 1fr)).
  • Tiles whose content is name-shaped (Leading, Top scorer, etc.) get grid-column: 1 / -1 at narrow widths via a container query, or use a CSS variable.
  • Iron rule for v2: any tile holding a player name must use a flexible grid cell — never a fixed-column flex item.
  • v2 mapping: TileGrid component in src/components/ with prop fullWidthSlots: ['leading', 'topScorer'].

Issue #7 — Missing density control

  • Modern data tools (GitHub, Linear, Notion, Airtable) all expose a density toggle. Reason: viewing tasks differs from scanning a leaderboard.
  • With 17 players in a league, a power user comparing PR across the field wants compact mode (10.5px font, 2px padding) to see everyone without scrolling.
  • A casual visitor checking on their own rank wants comfortable default for readability.

The fix

  • 3-state segmented toggle: compact / comfortable / spacious. Persist to localStorage["shabi-density"].
  • Implement via data-d attribute on the table → CSS selectors adjust padding + font-size.
  • Iron rule: density tokens are scoped — --row-h-compact, --row-h-comfortable, --row-h-spacious. Components read whichever the parent table specifies.
  • v2 mapping: DensityToggle primitive + density arg on every table preset. Default = comfortable.

Issue #8 — Chart hover affordance

  • Today's player chart sets a placeholder text "Hover or click a match to see details" but the details panel is rendered below the chart.
  • The user has to: hover bar 5 → look down 80px → read details → look back up → hover bar 6 → repeat. 12 round-trips for 12 matches.
  • Standard for 2026 (Posthog, Datadog, Grafana, Linear charts) is: tooltip floats next to the cursor.

The fix

  • Tooltip is a single position: absolute element inside the chart container. JS reads mouseenter on each bar, sets the tooltip's left to the bar's center, sets text, toggles [data-visible].
  • Tooltip is pointer-events: none so it doesn't steal hover from neighboring bars.
  • Caret arrow points down at the bar via ::after triangle.
  • v2 mapping: the ChartTooltip component already exists in the PLAN.md component list (Phase 3). This pattern is its acceptance criterion.

Issue #9 — Visual loudness in headers

  • Current top-bar: 64px height. Breadcrumbs below it: another 56px. Total: 120px every single page before any data renders.
  • On a 568px iPhone SE viewport, that's 21% of the screen dedicated to navigation chrome the user has already used to get here.
  • The gold-on-dark-green palette is correct for brand, but the height/density ratio is from 2018 not 2026.

The fix

  • Collapse to a 44px top-bar (the iOS/macOS sweet spot) with breadcrumb inline as quiet text.
  • Logo on left → breadcrumb in middle (font-weight: 500, opacity: 0.75) → actions on right.

Issue #10 — Keyboard navigation & ARIA

  • Today's tables have some keyboard support via tabindex but no visible :focus-visible ring — keyboard users get no signal where they are.
  • Sortable columns don't render a visual sort indicator (▲/▼) even when aria-sort is set in the DOM — screen-reader-only.
  • No vim-style row navigation (J/K) like GitHub / Linear / Superhuman.

The fix

  • Visible focus ring: tbody tr:focus-visible { outline: 2px solid var(--brand); outline-offset: -2px }. Plus an inset-shadow stripe on the leading edge so it's obvious even on row tints.
  • Sort indicators via CSS attribute selectors: th[aria-sort="ascending"]::after { content: " ▲" }. The DOM stays the source of truth — CSS just reveals it.
  • J/K hotkeys wired at the table level via keydown listener — focus moves to next [tabindex="0"] row.
  • ? key opens a shortcut overlay listing all bindings.
  • v2 mapping: shared useTableKeyboard.js hook applied by MFTable/SFTable mount fns. Axe-core test in tests/a11y/: every table must pass keyboard + aria-sort.
